# Findings

## Session 2026-03-25

### Offline flow
- `pipline_modified.py` is the dataset builder. It converts raw trade/orderbook inputs into 1-minute OHLCV plus microstructure features, scanner events, and forward-path columns used later for labeling.
- `quant_modeling.py` is the shared modeling core. It defines `BASE_FEATURES`, directional feature transforms, data-root resolution, dtype optimization, and the forward-path label cache via `build_labeling_cache`.
- Labels are not precomputed as fixed targets in the dataset. Instead, training/search scripts derive labels from future path columns using `build_labels(...)`, which means training behavior depends on runtime config (`horizon`, `tp_mult`, `sl_mult`, `label_mode`, `same_bar_policy`).
- `tune_all_modified.py` performs a quick parameter sweep on `horizon/TP/SL` with a simple chronological split and a fixed XGBoost template, then writes only the best config to `best_config.json`.
- `train_xgb.py` loads `best_config.json`, rebuilds labels, computes realized PnL in R units, applies class weighting, and trains the final XGBoost classifier.

### Offline validation details
- `train_xgb.py` supports two modes: standard `70/15/15` chronological split with purge gaps, and `--live` mode that removes the test holdout and reselects threshold on the last validation block before updating `best_config.json`.
- Threshold selection is explicitly profit-driven. `evaluate_thresholds(...)` scores candidate thresholds using realized PnL in R units, and `select_threshold(...)` picks the best smoothed validation profit subject to a minimum trade count.
- `robust_oos_search.py` provides a stronger walk-forward check than the basic training script: multiple chronological folds, purge gaps, threshold selection on validation, then fixed-threshold evaluation on subsequent test windows.
- Current `best_config.json` is tuned for `flow_reversal`, `first_touch`, `h=10`, `tp=3.0`, `sl=1.5`, with a global threshold plus separate long/short thresholds and reported strong aggregate OOS metrics.
- The reported search metrics are internally generated by this codebase; they are not yet evidence that live routing, latency, slippage, or exchange-state edge cases are handled.

### Live flow observations so far
- `live/main.py` wires a full event loop: aggregate ticks into bars, merge orderbook features, compute features, evaluate scanner, delay for event-alignment, then call the model and optionally place orders.
- `live/feature_engine.py` intentionally reuses `pipline_modified.add_features()` to preserve feature parity between offline and live minute-bar features. This is a strong design choice.
- `live/config.py` loads `best_config.json` at import time and exposes runtime thresholds / TP / SL / scanner params. Live behavior is therefore tightly coupled to the latest offline search artifact.
- The live scanner implementation is not generic. `live/scanner.py` only implements `FlowReversalScanner`, so the runtime currently matches only that scanner family even though offline code supports several scanners.
- `live/main.py` adds an extra post-trigger delay (`POST_WINDOW_S`) and event-aligned features before inference. This means live inference logic is more complex than a simple end-of-bar classifier call.

### Inference and execution observations
- `live/model_inference.py` loads feature names directly from the saved model and builds the input row in that order. This reduces column-order bugs and allows missing live features to flow through as `NaN`.
- Directional/event-aligned feature preparation in live inference is implemented manually inside `_prepare_features(...)`, not by reusing the shared modeling helper. Some fields overlap with `quant_modeling.add_directional_features`, but the logic is duplicated.
- `live/execution.py` is a real Gate.io REST client, not a paper-trading stub. It can fetch instruments, set leverage, place market orders, place separate TP/SL trigger orders, and query order history.
- `live/main.py` records signals/trades and stores TP/SL locally in `AppState`, but the correctness of ongoing state still depends on exchange push handling and startup reconciliation.

### Production-readiness risks
- Warmup uses REST candles only and fills microstructure fields with neutral placeholders (`aggressor_ratio=0.5`, `net_taker_vol_ratio=0.0`, `trade_gini=0.0`, etc.). Early live features therefore depend on synthetic history rather than true trade/orderbook history.
- Foreign-position handling is incomplete. The state model defines `has_foreign_position` / `foreign_positions`, but those fields are effectively unused in trading guards. Signal blocking checks only `state.open_position`.
- `_process_position_data(...)` distinguishes `"system"` vs `"foreign"` positions only in the dashboard payload. It does not convert foreign positions into a blocking state, so the strategy may open a new trade while an external/manual position already exists.
- Live scanner and event-aligner are specialized for `flow_reversal` with `bar_close` trigger semantics. If offline best config changes scanner family or trigger semantics, the live app will not remain aligned.
- Some live logic intentionally differs from backtest assumptions and is documented by test scripts, especially around event-aligned features and horizon-close behavior. Those differences are recognized by the repo, but not fully eliminated.
